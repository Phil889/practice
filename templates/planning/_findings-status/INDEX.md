# Findings Status — Index

**Purpose:** one-line-per-finding lookup so the supervisor and the orchestrators can answer questions like *"did we ever ship a fix for X?"* or *"what's open in the audits module?"* without loading every individual status file.

**Maintained by:** the supervisor (during `mode: hygiene`) + the build-loop (when a new finding-id is briefed) + the tester (when a verdict is appended).

**Format:** strict tab-aligned table for fast grep. The orchestrators' parsers depend on it.

## Status legend

- **BRIEFED** — orchestrator wrote the brief; implementer hasn't been dispatched yet
- **IN-PROGRESS** — implementer dispatched; commit not yet on disk
- **SHIPPED-PASS** — implementer committed + tester PASS
- **SHIPPED-FAIL-RETRY-N** — tester FAILed; on retry attempt N (cap 2)
- **SHIPPED-PASS-RETRY-N** — passed on retry attempt N
- **ESCALATED** — retry-cap exceeded; needs human review
- **DEFERRED** — orchestrator decided not to ship this round (logged in build summary)
- **ALREADY-FIXED** — implementer ran pre-fix probe, found GREEN state already, marked as previously-fixed (no commit)
- **ABANDONED** — orchestrator-decided not to ship at all (logged with reason)
- **ARCHIVED** — moved to `_archive/findings/<YYYY-MM>/` per the hygiene policy. Path field points to the archive file.

## Why this index exists

The `_findings-status/` directory grows with every shipped finding. After 6 months of `/audit-and-ship` weekly runs, that's ~50–200 files. Without an index, answering "what P0s are still open in module X?" means reading every file. The index keeps that lookup at <1K tokens.

The index is also the **bridge** between work that's in-flight (active files) and work that's settled (archived files). When a module-deep-dive needs to ask *"have we fixed any of the patterns the synthesis is about to cite?"* — it grep this index.

**Index entry conventions:**

- One row per finding-id (never duplicates)
- `Brief` column points to the brief file (immutable after build-loop writes it)
- `Status file` column points to the live status (or `—` if not shipped)
- For archived findings, the `Path` column overrides Brief + Status file with the archive location
- The `Title` column is the one-line headline; long descriptions stay in the status file

**Don't edit this file by hand.** It's regenerated by the supervisor on hygiene runs and appended to by the build-loop on every brief. Manual edits get overwritten.
